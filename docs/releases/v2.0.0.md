# Hacker Bob v2.0.0 Release Notes

`v2.0.0` realizes the frontier-topology hypergraph against the live Hacker Bob codebase. The session model is no longer a phase-ordered FIFO mutating a single `state.json`; it is four append-only planes — governance, frontier, scheduler, and claim — coordinated by a six-state lifecycle FSM. The release replaces every public identifier on the v1.x surface (MCP server name, tool prefix, session root path, finding vocabulary) with a target-state equivalent, with a one-release alias layer so existing integrations keep working through the v2.0 → v2.1 deprecation window.

## Breaking changes

### 1. MCP server name: `bountyagent` → `hacker-bob`

- `.mcp.json` entries are now written under `mcpServers["hacker-bob"]`. The MCP transport advertises `hacker-bob` as the product identifier on the wire.
- The tool-permission prefix on the host side is `mcp__hacker-bob__`.
- Existing v1.x installs are auto-migrated on the next install or update via the migration shim in `scripts/merge-claude-config.js`. The shim rewrites the legacy `mcpServers.bountyagent` server key to `mcpServers["hacker-bob"]` in `.mcp.json` and rewrites `mcp__bountyagent__*` permission strings to `mcp__hacker-bob__*` in `.claude/settings.json`. Operator-managed sibling servers and custom permission strings are preserved. The migration is idempotent: re-running install on an already-migrated workspace is a no-op for the rename surfaces.

### 2. Tool prefix: `bounty_*` → `bob_*`

- Every primary tool now declares a `bob_*` name. The complete rename is materialized at registry boot in `mcp/lib/tool-registry.js`.
- v1.x callers that hard-coded `bounty_*` names keep working through alias entries declared on each tool module (`aliases: ["bounty_*"]`). Invoking an alias appends a `governance.tool_deprecated` event to `session-events.jsonl` when a `target_domain` is in scope, with payload `{ tool, replacement, rename_cycle: "P.1", removal_release: "v2.1.0" }`.
- The alias layer ships in v2.0.0 and is removed in v2.1.0. No alias appears in the public `tools/list` catalog; only the canonical `bob_*` names are surfaced.
- Two compatibility shims persist as their own modules (`bounty_transition_phase`, `bounty_report_written`) and route to their lifecycle replacements (`bob_advance_session`, `bob_finalize_report`); these too are removed in v2.1.0.

### 3. Session root path: `~/bounty-agent-sessions/` → `~/hacker-bob-sessions/`

- New session writes target `~/hacker-bob-sessions/<target_domain>/`.
- On first access of any legacy `~/bounty-agent-sessions/<target_domain>/` directory, Bob **copies — never moves —** the directory into the canonical root. The legacy directory remains on disk as a fallback.
- Explicit cleanup is reserved for the v2.1.0 `--purge-legacy-session-root` flag. Operators that want to free disk space before v2.1.0 can delete the legacy root manually after confirming new sessions reach the canonical path.

### 4. Lifecycle states (FSM topology)

The eight-phase FSM `SURFACE_DISCOVERY -> AUTH -> EVALUATE -> CHAIN -> VERIFY -> GRADE -> REPORT -> EXPLORE` is retired in favor of a six-state lifecycle FSM:

```text
SETUP -> OPEN_FRONTIER -> CLAIM_FREEZE -> VERIFY -> GRADE -> REPORT
```

- `OPEN_FRONTIER` is re-entrant from every later state. `REPORT -> OPEN_FRONTIER` is authorized for follow-up discovery.
- The lifecycle tool is `bob_advance_session(target_domain, to_state)` with hard `allowedTransitions` in `mcp/lib/lifecycle-gates.js`.
- The v1.x tool `bounty_transition_phase(target_domain, target_phase)` remains as a deprecation shim that routes to `bob_advance_session` and emits a `governance.tool_deprecated` event. It is removed in v2.1.0.
- Each lifecycle transition emits a `governance.lifecycle.advanced` event to `session-events.jsonl` with `{from_state, to_state, prior_nucleus_hash, nucleus_hash}`. Nucleus hashes provide a timeline audit; re-entries of the same state can legitimately reuse a prior nucleus hash.

### 5. Claim vocabulary

- Per-record evidence is now `CandidateClaim` written to `claims.jsonl` (replacing `Finding` and `findings.jsonl`).
- Cluster grouping: `ClaimCluster` records in `claim-clusters.jsonl`.
- Frozen verification batches: `ClaimFreeze` artifacts under `claim-freeze/<claim_freeze_id>.json`, each identified by a content-addressed `claim_freeze_id`. Subsequent freezes produce new artifacts rather than mutating existing ones.
- Verification artifacts: `VerificationResult[]` records run against the frozen `ClaimFreeze` payload, not against a live re-read of the claim ledger.
- Report artifacts: `ReportSnapshot` records in `report-snapshots.jsonl` replace the prior `report.md` file-presence event. The lifecycle tool `bob_finalize_report` is hash-bound: it refuses to finalize unless all five upstream hashes (`claim_freeze_hash`, `final_verification_hash`, `evidence_hash`, `grade_verdict_hash`, `report_content_hash`) resolve, and the appended ReportSnapshot row binds the same five hashes plus a `snapshot_hash` for the row itself.
- The legacy `findings.jsonl` writer is preserved read-only through v2.0.x for back-compat and removed in a later release.

### 6. Frontier ledger

- `frontier-events.jsonl` is authoritative for discovery progress. Surface seed, lead enqueue, observations, closures, and blockers are all events on this ledger.
- `surface-index.json` and `task-queue.json` are materialized views over the frontier-events ledger.
- The legacy projection arrays on `state.json` (`terminally_blocked`, `explored`, `lead_surface_ids`) continue to populate during the dual-write window; they are removed in a follow-up cycle (D.3) once consumers have migrated to `frontier-projections.js` reads.

### 7. Verification round and repo-init schema tightening

- `bob_write_verification_round.results[]` now requires the complete per-claim tuple: `finding_id`, `disposition`, `severity`, `reportable`, `reasoning`, `repro_steps`, and `evidence_refs`. This is enforced by `mcp/lib/tools/write-verification-round.js`; callers that previously omitted replay steps or evidence references must send explicit arrays.
- `bob_init_session` is the web-mode entrypoint only. Passing `target_repo` now fails with `INVALID_ARGUMENTS` and `redirect_to_tool: "bob_init_repo_session"` from `mcp/lib/tools/init-session.js`. Repo-mode callers must use `bob_init_repo_session` so the repository target binding and authority record are created by the repo bootstrap path.

## Migration notes for downstream consumers

### Tool permission strings

Update any hard-coded `mcp__bountyagent__bounty_*` permission strings to `mcp__hacker-bob__bob_*`:

```diff
- mcp__bountyagent__bounty_http_scan
+ mcp__hacker-bob__bob_http_scan

- mcp__bountyagent__bounty_record_finding
+ mcp__hacker-bob__bob_record_candidate_claim
```

The install/update migration shim auto-rewrites legacy permission strings in `.claude/settings.json`. External configs that reference the legacy prefix should be updated to the canonical form. The `bounty_*` tool name aliases continue to resolve via the registry alias layer and emit a `governance.tool_deprecated` event each time they are invoked; the alias layer is removed in v2.1.0.

### Lifecycle-driven scripts

If you scripted phase transitions via `bounty_transition_phase`, switch to `bob_advance_session` and the lifecycle state enum:

```diff
- bounty_transition_phase(target_domain, "EVALUATE")
+ bob_advance_session(target_domain, "OPEN_FRONTIER")

- bounty_transition_phase(target_domain, "CHAIN")
+ # CHAIN is no longer a phase. Chain attempts are recorded via
+ # bob_write_chain_attempt during OPEN_FRONTIER / CLAIM_FREEZE.

- bounty_transition_phase(target_domain, "VERIFY")
+ bob_advance_session(target_domain, "CLAIM_FREEZE")  # freeze the claim batch
+ bob_advance_session(target_domain, "VERIFY")        # then advance to verify
```

### Session storage

Treat `~/bounty-agent-sessions/` as read-only. Bob copies legacy domain directories into `~/hacker-bob-sessions/` on first access; do not write to the legacy root yourself. Use the v2.1.0 `--purge-legacy-session-root` flag for explicit cleanup once you have confirmed that no agent or pipeline is reading from the legacy path.

### Findings → CandidateClaims

If your downstream pipeline read `findings.jsonl` directly, switch to `claims.jsonl` (`CandidateClaim` shape) and to the read tool `bob_read_candidate_claims`:

```diff
- # Read findings via the legacy ledger
- cat ~/bounty-agent-sessions/<domain>/findings.jsonl
+ # Read candidate claims via the MCP tool (authorized, schema-validated)
+ bob_read_candidate_claims({ target_domain: "<domain>" })
```

The CandidateClaim shape preserves every field on the legacy Finding record plus the new claim-plane metadata (`claim_id`, `cluster_id`, `claim_freeze_id` when frozen). The MCP tool is the supported read path; direct ledger reads remain possible during v2.0.x but lose the validation/migration semantics the tool provides.

### Frontier projections

If your tooling read `state.terminally_blocked`, `state.explored`, or `state.lead_surface_ids` directly off `state.json`, switch to the frontier-events ledger or to the materialized views (`surface-index.json`, `task-queue.json`). The legacy projections continue to populate through v2.0.x; the projection writers are removed in cycle D.3.

## Alias schedule

| Surface                               | v2.0.0 behavior                                   | v2.1.0 behavior |
|---------------------------------------|---------------------------------------------------|-----------------|
| `bounty_*` tool aliases               | Resolve via registry, emit deprecation event     | Removed         |
| `mcp__bountyagent__*` permission strings | Auto-rewritten to `mcp__hacker-bob__*` on install/update via migration shim | n/a (rewritten in v2.0.0) |
| `mcpServers.bountyagent` key in `.mcp.json` | Auto-rewritten to `mcpServers["hacker-bob"]` on install/update via migration shim | n/a (rewritten in v2.0.0) |
| `bounty_transition_phase` shim        | Routes to `bob_advance_session`                  | Removed         |
| `bounty_report_written` shim          | Routes to `bob_finalize_report`                  | Removed         |
| `~/bounty-agent-sessions/` legacy root | Read-only fallback; copied to canonical on access | `--purge-legacy-session-root` flag enables cleanup |
| `findings.jsonl` legacy writer        | Read-only fallback                                | Removed in a follow-up release |
| `state.terminally_blocked` / `state.explored` / `state.lead_surface_ids` | Dual-write with frontier-events | Removed by cycle D.3 |
| Eight-phase FSM (`PHASE_VALUES`)      | Backed by the lifecycle FSM via the shim          | Removed by cycle D.1 |

## Upgrade

```sh
npx -y hacker-bob@2.0.0 install /path/to/your/project
npx -y hacker-bob-cc@2.0.0 install /path/to/your/project
npx -y hacker-bob-codex@2.0.0 install /path/to/your/project
```

After upgrade, fully restart the host CLI in the project directory so it reloads `.mcp.json`, settings, hooks, and skills. The `hacker-bob` MCP server replaces the v1.x `bountyagent` entry on first install or update; the migration shim handles existing installs automatically and preserves operator-managed entries.
