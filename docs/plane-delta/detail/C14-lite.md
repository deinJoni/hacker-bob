**Kind / Tier / Type:** C / Δ1 / adopt (adopt-extend)   **Verdict from verification:** split — C14 splits into **C14-lite** (Δ1, this node: CWE/CVSS/refs prose, adopt-extend of the SC CWE table) and the net-new C14-bundle (Δ2). ## Spec (one-paragraph contract)
A **prompt-only** extension of the OSS-repo branch of the report-writer role (`prompts/roles/reporter.md:28-33`) that makes every reportable OSS finding emit three disclosure-form fields the web/SC branches already emit: a **CWE** id (table-driven from the finding's impact class), a **suggested CVSS-4.0 base vector** (assembled from the authoritative final-verifier `severity` plus the reachability classification the OSS branch already mandates), and a **References** line (CWE URL + affected file/commit permalink + any known upstream CVE/GHSA). Inputs: the same reader path the reporter already has — `bob_read_verification_round(round="final")` (severity, reportable) and `bob_read_candidate_claims` (impact/reachability prose). Output: three extra prose lines inside the existing OSS finding section — no new structured field, no `compose-report.js` change. Predecessors: none. Unlocks: `disclosure-friction-cut`. Enforced context budget: lives inside the existing per-section prose cap of **4096 chars** (`compose-report.js:64`) and the **600-word** per-finding ceiling (`reporter.md:109`); the added prompt block is capped at **~900 tokens**.

## What it does / holds
This is a **capability**, and the finding it produces is one a triager (and a GHSA / NVD intake form) recognizes without re-deriving: an OSS finding section that currently reads `file_path → symbol → repro → reachability → impact → remediation` (`reporter.md:30-33`) gains, before remediation:

- `CWE: CWE-787 (Out-of-bounds Write)` — looked up from a fixed impact-class→CWE table the slice adds, mirroring the SC table at `reporter.md:49-67` and the web `- CWE` line at `reporter.md:50`. Table rows map the exact OSS impact vocabulary the branch already names at `reporter.md:32` ("memory corruption, denial of service, arbitrary file/path effect, secret exposure, authz bypass, supply-chain compromise"): memory-corruption/OOB-write → CWE-787, OOB-read → CWE-125, use-after-free → CWE-416, DoS(loop/recursion/abort) → CWE-835/674/400, path traversal/arbitrary file effect → CWE-22, OS command injection → CWE-78, hardcoded secret → CWE-798, authz bypass → CWE-862, supply-chain (plaintext-HTTP/mutable-ref fetch) → CWE-494, integer overflow → CWE-190. (These are the exact CWEs the operator's own OSS campaign filed — CWE-787/125/416 stable-diffusion.cpp, CWE-494 libarchive/libredwg, CWE-22+CWE-78 gpac, CWE-798 netatalk.)
- `Suggested CVSS-4.0: CVSS:4.0/AV:L/AC:L/AT:N/PR:N/UI:P/VC:N/VI:N/VA:H (triager should re-score)` — magnitude anchored by the final-verifier `severity` band, with `AV` derived from the reachability class the reporter is already required to write at `reporter.md:31` (network/protocol message → `AV:N`; local file/CLI parse → `AV:L`; CI/build event → `AV:L`), and `UI`/`PR` set from whether maintainer/user action is named in that same reachability sentence.
- `References: https://cwe.mitre.org/data/definitions/787.html ; <repo permalink to file:line> ; <upstream CVE/GHSA if surfaced via public-intel>`.

## Failure mode it prevents
The specific bad outcome: OSS disclosure reports ship **without** the two fields the disclosure channel demands, so the operator hand-assembles a CWE id and a CVSS vector at filing time for every GHSA/email (the friction observed across the netatalk/openexr/exiv2/libredwg filings). Worse, an ad-hoc score invented at submission time risks **contradicting the authoritative final-verifier severity** — the exact authority inversion `reporter.md:108` and the SC branch (`:48`) exist to forbid. C14-lite makes the metadata fall out of MCP-owned severity deterministically, so the report is GHSA-paste-ready and internally consistent.

## Risks & doctrine compliance
- **substrate-as-trajectory:** N/A — no substrate added; this consumes shipped reader tools.
- **index-without-consumer:** N/A — the consumer (the report section) is the same artifact being extended; the metadata has an immediate reader (the triager / GHSA form).
- **context-as-strategy:** avoided — this does not pretend a prose table is a verification primitive; it is explicitly descriptive disclosure metadata, gated behind the already-verified reportable finding.
- **auto-kill / freeze-then-verify inversion:** none — C14-lite adds no kill/suppress path; it only annotates findings that already passed the reportability gate (`reporter.md:6-7`).
- **Severity-authority risk (the one real hazard):** a free-typed CVSS could imply a severity that contradicts the final-verifier value — forbidden by `reporter.md:108`. Mitigated by anchoring the vector to the authoritative `severity` band and labeling it `(suggested — triager should re-score)`, honoring the reporting guardrail against speculative/overstated impact.
- **Honest correction to the node intent (flag).** The intent says "assemble CVSS from final-verifier severity + `attack_vector`." Code check: `attack_vector` is a **surface/evaluator-only** field (`mcp/lib/assignment-brief.js:139`, AV:N-vs-AV:L triage), and is **not** in the reporter's read path — the verification round carries only `disposition/severity/reportable/reasoning` (`mcp/lib/tools/write-verification-round.js:51-72`), and candidate claims carry `severity` but no `attack_vector` (`mcp/lib/claims.js:292-316`). So the reporter cannot literally read the surface `attack_vector`. The grounded, prompt-only substitute is to derive the CVSS `AV` metric from the **reachability classification the OSS branch already requires at `reporter.md:31`**, anchored in magnitude by the final-verifier `severity`. This keeps the node prompt-only and inside the reporter's existing reader path; promoting the surface `attack_vector` into the reporter context would be a brief/schema change and belongs to the Δ2 C14-bundle, not the lite slice.
