# I13 — Cross-target transfer index

**Kind / Tier / Type:** I (pillar / pre-computed index) · Δ4 · net-new   **Verdict from verification:** revise — net-new node kept, but its predecessor was repointed. WEAVE.md:15 + the structural-regression headline (WEAVE.md:33) confirm `findings-index.js` was deleted in commit `c02179c`; the original "cheap because the index already exists" rationale is dead. The cross-target retrieval-into-brief must be **rebuilt on the claims subsystem**, and the one surviving cross-target primitive (`mergePriorClaimMatches`) is an orphaned producer that this node finally gives a consumer.

## Spec (one-paragraph contract)
**Inputs:** a confirmed finding-family on target A — a set of `CandidateClaim` rows (`readCandidateClaims(domainA)`, `mcp/lib/claims.js:456`) or a frozen cluster `CC-*` (`readClaimClusters(domainA)`, `mcp/lib/claim-clusters.js:94`) — plus, per authorized target B, B's persisted `symbol-surface-index.json` (`readSymbolSurfaceIndex`, `mcp/lib/symbol-surface-index.js:148`) and `surface-graph.jsonl` (`queryEdges`/`neighbors`, `mcp/lib/surface-graph.js:161,197`). **Output:** a deterministic, content-addressed set of *sibling-hypothesis* records (one per structurally-similar surface on B), each shaped `{target_domain, claim_id, similarity, structural_match_reasons[], surface_id_on_b, prior_claim_summary}`, merged/ordered via `mergePriorClaimMatches` (`claim-clusters.js:115`) and rendered into a bounded `transfer_slice` for B's evaluator brief. **Predecessors:** I1 (surface graph), I3 (symbol-to-surface index), `claims.js`, `claim-clusters.js`. **Enforced context budget:** the index is index-only and read-only over already-persisted per-target artifacts — it never issues HTTP and never crosses into a target's live surface; the brief slice is capped at 4096 chars (see Context budget). The cross-target scan window reuses the 200-domain saturation ceiling (`CROSS_SESSION_ANALYTICS_MAX_SESSIONS = 200`, `mcp/lib/pipeline-analytics.js:41`) — which is exactly the window the `mergePriorClaimMatches` saturation-guard comment describes (`claim-clusters.js:109-114`).

## What it does / holds
**Stores / computes.** For a confirmed family on A, it derives a *structural transfer key* per claim from the claim's `surface_ids` (`claims.js:266,301`): dereference those ids through A's `by_surface` map (`lookupBySurfaceId`, `symbol-surface-index.js:172`) to recover the surface footprint — `{path, method, framework, handler_hint, edge_kind}` (`symbol-surface-index.js:102-111`) — and pull related `tech` / `auth_scheme` / `endpoint` neighbors from A's surface graph (`summarizeSurfaceGraphForSurface`, `surface-graph.js:230`). For each authorized B it scores structural similarity against B's index: normalized route-shape equality (the placeholder-collapsing `normalizeRoutePath`, `symbol-surface-index.js:37` — `/users/:id` ↔ `/users/{id}` ↔ `/users/<int:id>` all reduce to one slot), framework match, method match, and tech/auth-scheme overlap from B's surface graph. Each scored hit becomes a record carrying `target_domain`, `claim_id` (the A-claim's `CL-*` id, `claims.js:239`), and a numeric `similarity` — the exact triple `mergePriorClaimMatches` keys and sorts on (`claim-clusters.js:103-133`).

**Bounded query it enables.** "Given confirmed family F on A, what are the top-K structurally-similar surfaces on each authorized B, same-target-first?" — answered as a ≤4096-char slice, not by stuffing A's and B's raw artifacts into one prompt.

## Failure mode it prevents
Two, concretely. (1) **Experience amnesia across targets:** without it, a confirmed BAC/IDOR/SSRF family found on A is invisible when Bob later evaluates B, so Bob re-derives the same hypothesis from zero — the failure I6 was meant to stop, now reopened by the `c02179c` deletion (WEAVE.md:33,119). (2) **Hallucinated cross-target correlation:** the naive alternative — dropping A's findings and B's surface dump into one context — produces invented "this looks like that" links with no structural basis. This index forces every sibling hypothesis to cite a concrete `structural_match_reasons[]` (shared normalized path-shape, framework, method, tech) anchored in content-addressed index entries, so a triager can check the structural claim instead of trusting an LLM's vibe.

## REVIEW (parishioner / real-target gate)
On two authorized targets B and C (same program / same operator authorization), seed a confirmed finding-family on A, then run an evaluate on B. The gate closes when ≥1 sibling hypothesis surfaced from the `transfer_slice` is **independently confirmed on B** as an actionable finding a triager would accept (not a duplicate of an A artifact, not informational) — i.e., transfer produced a real B-finding that a from-scratch B evaluate measurably reached slower or missed. **Honest caveat:** this is a long-horizon gate — it needs two authorized targets that genuinely share structure plus a pre-confirmed A-family, which is rarer than a single-target run. Until that alignment exists, the closeable interim gate is engineering/fixture-only: the determinism canary + the `mergePriorClaimMatches` saturation-ordering test passing on fixtures. Do not mark the parishioner gate closed on fixtures.

## Context budget
- **`transfer_slice` per brief:** ≤ 4096 chars (~1k tokens) — half of `surface_graph_slice`'s 8192 (`assignment-brief.js:278`), since transfer is a supplement, not the primary surface view. Mirrors the deleted I6 priors-injection intent (≤2k tokens at evaluate start; per-prior ≤200 tokens, doc line 218).
- **Per-sibling-hypothesis entry:** ≤ 200 tokens (id pointer + structural reasons + one-line sanitized summary).
- **Sibling count K:** 5 default, 10 hard ceiling (tracks I1's `SURFACE_SLICE_DEFAULT_LIMIT = 5` / `MAX_LIMIT = 25`, `surface-graph.js:217-218`).
- **Cross-target scan window:** ≤ 200 domains (`CROSS_SESSION_ANALYTICS_MAX_SESSIONS`, `pipeline-analytics.js:41`).
- No new long-lived subagent is introduced — this is an index consumed at brief-construction time, so there is no per-subagent budget beyond the slice itself.

## Risks & doctrine compliance
- **index-without-consumer (the primary trap here):** I6's deletion proves the danger — `mergePriorClaimMatches` is a producer with no consumer *right now*. This node is only legitimate if it ships *with* C13 consuming the slice; an I13 that builds a slice nothing injects would repeat the exact regression verification flagged. Build slice 5 (brief wiring) in the same PR series, not later.
- **substrate-as-trajectory:** the index must not become a parallel finding store. It reads `claims.js`/`claim-clusters.js` as source of truth and emits a derived, content-addressed slice — it does not author or mutate claims, freeze state, or advance any phase.
- **context-as-strategy:** transfer hypotheses are *leads*, not findings. The slice seeds B's evaluator; impact still requires live confirmation on B under B's own scope. Never let a transferred prior shortcut the validate-before-record rule (hunting.md).
- **auto-kill / freeze-then-verify:** this node introduces no termination logic and must not. A low similarity score may not auto-suppress a surface on B — it only ranks; B's own evaluation decides. No premature termination of a B surface on the basis of an A prior.
- **Scope / no-PII / network policy:** cross-target reads are strictly index-only over persisted artifacts; **no HTTP crosses target boundaries** (consistent with the scoped-MCP-HTTP rule in hunting.md and Plane-O `--network none` discipline). `target_url_policy: index_only_no_target_url_export` (`session-authority.js:352`) plus `validateNoSensitiveMaterial` (`sensitive-material.js:73`) enforce that A's URLs/secrets/operator identifiers never land in B's brief. The hard scope-auth gate (`H-C13`: B must be `bob_init_session`+`bob_start_wave` authorized) is a C13/dispatch responsibility, restated here so I13 is never wired to emit slices for unauthorized B.
- **(assumed) Multi-VM reach for C16:** I3's `symbol-surface-index` is built from web route records + `attack_surface.json` (`symbol-surface-index.js:88-93`, doc line 119) — it is web-route-shaped. The `H-C16` fan-in (cross-VM, with I5+I7) assumes a VM-neutral structural key exists; I could **not** verify a non-web surface footprint in code. Treat cross-VM transfer as out of scope for I13's first ship and a separate keying problem, not a free extension.
