# C15 — Adversarial self-patch loop

**Kind / Tier / Type:** C (capability) / Δ4 / net-new   **Verdict from verification:** **keep** (largest-effort node in the C-cluster; one doctrine guardrail must be encoded in code, not just prose — `verification/verify-C.md:49-55`).

## Spec (one-paragraph contract)
Given a single **confirmed, reportable OSS-repo finding** (final-verifier `reportable: true` against a frozen `claim_freeze_id`) whose PoC is a runnable in-container harness, the orchestrator runs a **bounded iterate-until-robust controller** that alternates two agents: the **chain-builder** proposes a minimal candidate patch (a local `git` diff) and, on each round, generates ≤K adversarial PoC *variants* that attack that patch; an **OSS evaluator-agent** applies the patch and replays the PoC/variants in-container (S14's `git apply` on the `read_write` `/work` mount via `bob_repo_docker_run`). Each apply→replay is recorded as a **C10 differential evidence row** `{vuln_run_id, control_run_id, control_kind: "self_patch", verdict}`. **Inputs:** one frozen reportable finding + its PoC harness, the repo session, a round cap. **Outputs:** (1) a `suggested_fix` that survived self-attack, attached **additively** to the finding (feeds C14/C17); (2) strengthened reachability evidence when a variant defeats a candidate patch. **Predecessors:** C10 (Δ-new), S14 (Δ-new). **Enforced context budget:** controller frontier ≤5k tokens; per-round agent briefs ≤8k each; round cap N≤4; per-round in-container stdout bounded by the existing 16 MB `repo-runs/` cap (`paths.js:323-327`). **HARD INVARIANT:** a patch that fixes the PoC must **never** auto-suppress the pre-patch finding — the controller has *no code path* to mutate the frozen `CandidateClaim`, verification round, or grade verdict.

## What it does / holds
The capability produces a finding a maintainer/triager recognizes as **a confirmed bug plus a patch that has been adversarially stress-tested**: "PoC fires at HEAD; here is a minimal fix; here are N self-generated variant exploits that the fix was shown to defeat; here is one variant that defeated an *earlier* candidate fix (which is why the final fix is shaped this way)." Concretely, per round it holds:
- A content-addressed candidate-patch diff (chain-builder proposal, bounded length).
- A C10 differential row per apply→replay: `vuln_run_id` (the HEAD run that fires the bug), `control_run_id` (the patched-tree run), `control_kind: "self_patch"`, `verdict ∈ {still_fires, no_longer_fires}` — written through C10's extended `bob_write_evidence_packs` (`mcp/lib/evidence.js:510 writeEvidencePacks`, `mcp/lib/tools/write-evidence-packs.js`).
- One chain-state-tree node per iteration (`bob_append_chain_node`, parent = prior round's `state_hash`), so the whole propose→attack→repatch lineage is replayable even though the LLM-generated variants are not byte-reproducible.
- A terminal robustness verdict (`robust` after a full round where no variant re-triggers, or `unresolved` at the round cap) recorded as a frontier event (`bob_append_frontier_event`) + the `suggested_fix` block surfaced into the OSS report branch (`prompts/roles/reporter.md:28-31`) as additive metadata.

It does **not** decide whether the finding is real — that was settled at VERIFY before the loop ever runs.

## Failure mode it prevents
Two:
1. **The thing the invariant exists to stop:** an agent "fixes" the bug, sees the PoC go green, and silently downgrades/closes the original finding — destroying a real, already-confirmed vulnerability report. The pre-patch vuln *is still real*; a fix is a remediation suggestion, never a disposition change (`verify-C.md:54`).
2. **The shallow-fix failure:** shipping a one-shot "suggested fix" in a disclosure that a maintainer immediately bypasses with a trivial input variant (the residual-fix pattern Bob keeps hitting — netatalk CVE-2026-44066 residual, libraw `unpack_thumb` residual of the 2020 commit). The adversarial loop forces the fix to survive Bob's *own* variant attacks before it goes in the report, turning "suggested fix" from a liability into corroboration of reachability.

## REVIEW (engineering)
- **Tests.** `test/self-patch-loop.test.js` (new): round-cap termination; convergence predicate; and — the load-bearing one — `assertNoSuppression` rejects every disposition-mutating action, and a fixture where a candidate patch makes the PoC pass asserts the finding remains in the final reportable set with `reportable: true` unchanged. Add a `record-self-patch-round` tool test (schema has no disposition field; row is content-addressed and idempotent on re-record). Extend `test/mcp-test-discovery.test.js` (S12) so the new tool↔guard-test coverage is registered (prevents a silent drop of *this* node).
- **Context-budget compliance.** Controller frontier ≤5k; per-round chain-builder and evaluator briefs ≤8k each; round cap N≤4 ⇒ ≤8 in-container runs/finding; patch-diff and variant payloads capped; in-container stdout bounded by the 16 MB `repo-runs/` cap (`paths.js:323`).
- **Determinism / content-hashing (S1).** Each candidate patch, each `control_run_id`, and each differential row is content-addressed; the per-(vuln_run_id, control_run_id) verdict is hash-stable. The adversarial *variant generation* is LLM-nondeterministic by nature — so the controller logic, the round verdicts, and the chain-state lineage must be deterministic and replayable from recorded nodes even though regeneration of variants is not byte-reproducible. State this explicitly in the test (assert lineage replay, not variant regeneration).
- **Failure-mode coverage.** Orphan: a half-run loop (patch applied, replay never recorded) must leave the finding untouched and recover via the chain-state frontier. Partial: round cap hit with no robust verdict ⇒ `unresolved`, finding still reported additively. Malformed: a patch that fails to `git apply`/build ⇒ record `blocked` (mirror chain-attempt `blocked` semantics), not a fire/no-fire verdict, and never a suppression. Replay-unsafe: in-container runs serialize through the existing docker replay lease (S3).
- **Slab invariants preserved (cite).** S1 content-addressed evidence; **S2 manifest-coupled state** — the loop must not advance lifecycle state or write partial state (it runs entirely outside the FSM transitions, like post-report evidence mode); **S3 per-pack replay leases** for in-container runs; and the blanket slab invariant (`capability-hypergraph.md:57`): no advancing state without manifest refresh, no artifacts without content-addressing, no bypassing replay-lease typing. Plus the S15/Plane-O substrate invariants inherited via S14: **O-P1** (init never clones) and **O-P3** (`--cap-drop ALL`, `no-new-privileges`, `USER 1000:1000`, `--network none`).

## REVIEW (parishioner / real-target gate)
Honest read: **this is Δ4, long-horizon, and gated behind C10+S14.** The closing engineering-fixture gate (checkable now, the one that matters for the invariant) is: *on a fixture finding whose PoC a candidate patch makes pass, the finding remains in the final reportable set and the report unchanged — the loop produced a `suggested_fix`, not a suppression.* The real-target value gate (long-horizon): on a real OSS finding of the residual-fix class Bob already encounters (libcupsfilters/gpac/netatalk-residual-shaped), the loop demonstrates **both** (a) a `suggested_fix` that survives ≥N self-generated variant attacks, and (b) at least one round where an *earlier* candidate patch was shown insufficient because a variant still fired — proving the loop adds value over a naive one-shot fix. A maintainer accepting the self-attacked fix as part of a disclosure is the ultimate parishioner signal but is downstream (C14) and not required to close C15.

## Context budget
- Controller frontier in orchestrator: **≤5k tokens**.
- Per-round chain-builder brief (propose / re-attack): **≤8k**.
- Per-round OSS evaluator brief (apply+replay): **≤8k**.
- Round cap: **N≤4**; variants per round **K≤3** ⇒ ≤ (1 + 3) replays/round, ≤16 in-container runs/finding total.
- Candidate-patch diff payload: **≤4096 chars** (mirror the chain-rollup narrative cap); in-container stdout/stderr bounded by the existing 16 MB `repo-runs/` cap (`paths.js:323-327`).
- `self-patch-rounds.jsonl` row: bounded, content-addressed, no raw secrets/PII.

## Risks & doctrine compliance
- **Auto-kill inverting freeze-then-verify / no-premature-termination — THE central risk.** A "patch fixes PoC" outcome must cap at `suggested_fix`, never a kill. Mitigations are layered and code-first: (1) the loop runs strictly *post-VERIFY* on a frozen, already-confirmed reportable claim — the disposition decision is already made and immutable; (2) it follows the established **post-report-evidence-mode precedent** (`orchestrator.md` REPORT: a mode that "must not update findings… verification, grade, or report artifacts"); (3) the round-writer tool (`bob_record_self_patch_round`) has no schema field that can change disposition, and `assertNoSuppression` blocks any routing of a self-patch result into claim/verification/grade mutators; (4) a guard test asserts a "fixed" PoC keeps the finding reportable. This is the same doctrine line C9 walks (`cap, never kill`).
- **substrate-as-trajectory:** clean — C15 is a capability that produces robust findings + suggested fixes, pulling directly on C10+S14; it is not speculative substrate.
- **index-without-consumer:** N/A (capability node); its own consumers are the report, C14 bundle, and C17 sentinel.
- **context-as-strategy:** clean — bounded per-round subagents + the chain-state-tree carry state; the controller never widens context to "hold the whole patch history."
- **Plane-O O-P1 / O-P3 (`--network none`):** the candidate patch is a **local diff**, so apply+rebuild+replay runs entirely under S14's existing `read_write` `/work` mount with `--network none` — no upstream fetch (that is C10's separate `upstream_fix` control_kind, which is the only path needing history; `self_patch` stays offline). State this distinction so the loop never reaches for network.
- **hard-rule no-PII:** in-container only; no target endpoint POSTs, no operator identifiers in any round row, diff, or frontier event.
- **Frontier-node safety/scope gate:** the loop only operates on an *already-confirmed, in-scope* finding on an authorized OSS target and generates no new traffic against any live deployed service (it runs against the repo in-container); it performs **no** cross-target dispatch (that is C13). Scope authorization is inherited from the finding under test — no new external attack surface is opened.
