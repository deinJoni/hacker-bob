# X8 — Live in-run observability (SSE)
**Kind / Tier / Type:** X (cross-cutting) / Δ4 (polish) / adopt (SSE *upgrade* over an existing poll).   **Verdict from verification:** **keep** (WEAVE.md:28; verify-X.md:21–28). Critic moved tier Δ2→Δ4 polish; scope-honesty edit folded in: the dashboard *already* serves a weak live view (`/api/snapshot` poll, `dashboard.js:482`), so this is a push-based upgrade, not greenfield. Design is *aligned* with `X-rej-4` (nodes.json:55) — it tails the existing `frontier-events.jsonl` rather than standing up an in-memory bus.

## Spec (one-paragraph contract)
A new read-only Server-Sent-Events route `GET /api/session/:domain/events` on the existing loopback dashboard server. **Inputs:** path `:domain` (validated by `assertSafeDomain`), optional `Last-Event-ID` request header (resume cursor), optional `?backlog=` cap. **Outputs:** a `text/event-stream` of merged, ts-ordered frames folded from two append-only ledgers in `sessionDir(domain)` — `frontier-events.jsonl` (`frontierEventsJsonlPath`, paths.js:116) and `pipeline-events.jsonl` (`pipelineEventsJsonlPath`, paths.js:112; carries `verification_written`/`wave_started`/`wave_merged`/`grade_written`/`report_written`, pipeline-events.js:28–52 — this is the "merge verification+wave ledgers" mapping). **Predecessors:** `dashboard.js` + the frontier/pipeline ledgers (all SHIPPED). **Enforced context budget:** introduces **zero** evaluator-brief slice and **zero** subagent, so there is no token budget; the only enforced bounds are a per-frame `data:` byte cap and a cold-start backlog cap (see Context budget). Loopback-only; pure observer (never writes session state).

## What it does / holds
It holds no new persisted state. It is a *fold* (in the `frontier-projections.js` sense, that file's header comment :18) of two existing append-only ledgers into one live, resumable stream the operator's browser consumes in place of the 5-second `/api/snapshot` poll.

Concretely, on connect it:
1. Validates `:domain` via `assertSafeDomain` (reused from frontier-events.js:60 / paths.js) so `:domain` cannot traverse out of `sessionDir`.
2. Reads both ledgers tolerantly (a partial trailing line is skipped, not thrown — unlike `readJsonlStrict`, fabric-common.js:113 which throws on malformed input).
3. Assigns each record a stable frame id: frontier records already carry `event_id` (`FE-<hash>`, frontier-events.js:52–54) and `event_hash` (:95); pipeline records lack a per-event id, so the reader derives a content-addressed `PE-<sha256(canonical)>` at read time (mirroring `generatedFrontierEventId`, frontier-events.js:52). Frame id = `${ts}|${source}|${recordId}`.
4. Merges by ordering key `(ts, source_rank, recordId)`, emits frames after the `Last-Event-ID` cursor (cold start: last `backlog` frames only), then polls the two files on a fixed interval and flushes newly-appended frames live.

The "finding a triager recognizes" framing does not apply — X8 is cross-cutting observability, not a capability that produces findings. Its product is an operator-facing live view (the node's only `unlocks`: `live-operator-view`).

## Failure mode it prevents
The operator flying blind during a long multi-wave run, or having to hammer `/api/snapshot` (whole-snapshot recompute via `buildDashboardSnapshot` → `readPipelineAnalytics`, dashboard.js:298–306) on a timer to notice that a wave merged, a verification round was written, or a finding landed. The poll model also silently loses mid-run transitions between polls and cannot re-attach with continuity. SSE with `Last-Event-ID` resume closes both gaps: a dropped connection (laptop sleep, tab reload) resumes exactly after the last frame the browser saw, with no duplicate and no gap (modulo ledger trim, handled below).

## Predecessors & unlocks
- **`dashboard.js` (SHIPPED).** `routeDashboardRequest` (dashboard.js:545) currently routes only `/`/`/index.html` (:552) and `/api/snapshot` (:556) with a 404 fallback (:565); `startDashboardServer` (:572) already creates the server (`http.createServer(...routeDashboardRequest...)`, :582) and already enforces the loopback warning (`isLoopbackHost`, :59; warning :574–580). The new route slots into the existing `routeDashboardRequest` switch — no new server.
- **`frontier-events.jsonl` (SHIPPED).** `readFrontierEvents` (frontier-events.js:108), stable `event_id`/`event_hash`, append-only with `FRONTIER_EVENTS_MAX_RECORDS = 20000` trim (:31).
- **`pipeline-events.jsonl` (SHIPPED, Δ-new wiring).** `PIPELINE_EVENT_TYPES` already includes every verification/wave/grade/report lifecycle event (pipeline-events.js:28–52). Reading it as a second SSE source is the net-new wire.
- **Unlocks:** `live-operator-view` only. No cross-tier dependency; this is the lowest-stakes, most self-contained node in the X cluster (verify-X.md:28,43).

## DO (build slices)
1. **Spec + tolerant tail reader (one commit).** New file `mcp/lib/dashboard-event-tail.js`. Export `readSessionEventFrames(domain, { sources })` that: calls `assertSafeDomain(domain)` (paths.js); reads `frontierEventsJsonlPath`/`pipelineEventsJsonlPath` (paths.js:116/:112) with a **tolerant** line splitter (split on `/\r?\n/`, `JSON.parse` each line inside try/catch, **skip** unparseable lines rather than throw — deliberately diverging from `readJsonlStrict`, fabric-common.js:113, to survive a partial trailing append from `appendJsonlLines`, storage.js:142); derives the per-frame id (`event_id` for frontier; synthesized `PE-<hash>` for pipeline via `hashCanonicalJson`, already imported in frontier-events.js:16); returns frames sorted by `(ts, source_rank, recordId)` with a per-frame field allowlist/compaction pass (mirror `compactDashboardSession`, dashboard.js:196, so no raw artifact body or secret-bearing field leaks — frontier/pipeline events carry only hashes like `nucleus_hash`/`egress_identity_hash`, pipeline-events.js:281–285, which are safe to surface). Unit-tested in isolation.
2. **SSE response helper (same or next commit).** In `dashboard.js`, add `openEventStream(req, res)` alongside `sendJson` (:349) / `sendHtml` (:360): `res.writeHead(200, { "content-type": "text/event-stream", "cache-control": "no-store", connection: "keep-alive" })` — **no `content-length`** (the existing helpers set it, :354/:364, which would break streaming). Add a `writeFrame(res, frame)` that emits `id: <frameId>\nevent: <source>\ndata: <json>\n\n` and a `writeHeartbeat(res)` that emits `: ping\n\n`.
3. **Wire the route (same commit as 2).** In `routeDashboardRequest` (dashboard.js:545), before the 404 fallback (:565), match `^/api/session/([^/]+)/events$`. Hard-gate to loopback: if `!isLoopbackHost(baseOptions.host)` (:59) reply `403 {error:"loopback_only"}` via `sendJson` (the route is an unauthenticated live tail of session internals, so it must *refuse* off-loopback, stronger than the server's existing warn-only posture, :574). Parse `?backlog=` with the existing `parseInteger` (:40) clamped to `MAX_BACKLOG`. Read `req.headers["last-event-id"]`. Open the stream; emit the resume backlog (frames after the cursor's `(ts, source_rank, recordId)` ordering key; cold start → last `backlog` frames); then start a `setInterval` poll (default 1000ms) that re-reads via slice 1 and flushes frames after the in-memory last-sent key. Register `req.on("close", …)` to `clearInterval` (no leaked timers) and a heartbeat `setInterval` (default 15000ms).
4. **Trim-safe resume (same commit).** Because `trimJsonlFile` (invoked from `appendJsonlLines`, storage.js:147–148) can shrink a ledger from the front at the 20000 cap, resume must filter by the `(ts, source_rank, recordId)` **ordering key**, never by line count. If the exact `Last-Event-ID` tuple is absent but surviving frames sort *older* than it, emit a one-shot `event: resync` comment frame so the client knows a gap occurred. Document this in the file header.
5. **Client upgrade (same commit).** In `renderDashboardHtml` (dashboard.js:370–543), replace the timer-less single `load()` (the `fetch("/api/snapshot", {cache:"no-store"})` at :482 stays as the cold snapshot) with an `EventSource("/api/session/<domain>/events")` that patches the rendered table/bottlenecks on each frame and auto-resumes (the browser sends `Last-Event-ID` natively on reconnect). Keep the existing `/api/snapshot` route untouched as the initial paint + fallback.
6. **Ship.** One PR. Regenerate nothing tool-side (no MCP tool, no role bundle, no skill change — this is a CLI dashboard route, not part of the MCP tool registry S5). Confirm `npm run check:syntax` and the dashboard test (slice below) pass.

## REVIEW (parishioner / real-target gate)
This is an **engineering/fixture gate, not a findings gate** — X8 produces no finding, so the anti-pattern #3 "capability without parishioner review" (capability-hypergraph.md:535) is satisfied by an operator-facing observability check, the same posture X3 took. Closing outcome: during a real authorized multi-wave `/bob-evaluate` run, open the dashboard, confirm the live view shows `wave_started` → `evaluator_stopped` → `wave_merged` → `finding_recorded` → `verification_written` → `grade_written` frames appear **without manual refresh**, reload the tab mid-run and confirm `Last-Event-ID` resumes with no duplicate/no gap, and confirm zero writes appear in `sessionDir` attributable to the dashboard (S2). No false-positive/triager-recognition axis applies.

## Context budget
- New evaluator brief slice: **none.** New subagent: **none.** Added LLM context: **0 tokens** (loopback HTTP route only).
- Per-SSE-frame `data:` payload: **2 KB hard cap**, truncate-and-flag.
- Cold-start backlog (`?backlog=`, no `Last-Event-ID`): default **200**, `MAX_BACKLOG` ceiling enforced by `parseInteger` clamp (dashboard.js:40).
- Heartbeat 15 s; poll cadence 1 s; both are constants, not per-run brief inputs.

## Risks & doctrine compliance
- **substrate-as-trajectory:** clean — no new substrate; it folds two *existing* shipped ledgers, and its consumer (the dashboard client, slice 5) ships in the same PR.
- **index-without-consumer:** N/A — no index built; the consumer is the live operator view.
- **context-as-strategy:** explicitly clean — zero added LLM context; the design *narrows* operator overhead (push vs poll) rather than widening any window.
- **auto-kill / freeze-then-verify inversion:** N/A — read-only observability touches no verification or termination decision; it cannot prune, kill, or short-circuit a claim. No `X-rej-1/2` surface.
- **`X-rej-4` (nodes.json:55) — the one live doctrine constraint:** honored by construction. No in-memory event bus; the SSE is a **file-backed tail over `frontier-events.jsonl`** (plus `pipeline-events.jsonl`), i.e. it obeys "frontier-events.jsonl already *is* the stream" (verify-X.md:27).
- **Loopback / scope gate:** the route hard-refuses off-loopback (`isLoopbackHost`, :59) because it is an unauthenticated live tail of session internals — stronger than the server's existing warn-only default (:574), and the right posture for streaming session state. No Plane-O `--network none` or no-PII concern arises (no target traffic, no operator identifiers ever enter the stream; the compaction allowlist keeps it to hashes + summaries).
- **Assumption flagged:** "merge verification+wave ledgers" is implemented via `pipeline-events.jsonl` (which carries `verification_written`/`wave_*`/`grade_written`, pipeline-events.js:28–52), *not* by tailing the discrete audit-graded verification-round/wave-handoff JSON files (paths.js:395–408). That is the doctrine-safe reading (those files are audit-graded and not append-only JSONL, so tailing them would need `fs.watch` + synthetic events and would read audit-graded artifacts). If a future requirement needs per-round *content* live, that is a separate, heavier node — not in X8's Δ4 scope.
