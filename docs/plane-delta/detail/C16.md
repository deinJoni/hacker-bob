# C16 — Cross-VM invariant/finding transfer
**Kind / Tier / Type:** C / Δ4 / net-new   **Verdict from verification:** ➕ new (verification-surfaced). Critic §2: "the 6-chain SC substrate had zero new frontier consumers." C16 is the consumer. Predecessor I13 was repointed off the deleted `findings-index.js` (I6) onto the claims subsystem (`claims.js` + `claim-clusters.js`); honor that — there is no embeddings index to lean on.

## Spec (one-paragraph contract)
**Inputs:** a confirmed invariant on one VM deployment — concretely an I5/`invariant-runner` record with `outcome: "test_passed"`/`"test_failed"` and a `run_hash` (`mcp/lib/invariant-runner.js:656-663`) plus its `vulnerability_class` (`mcp/lib/invariant-template-corpus.js:144-159`); and a set of **independently scope-authorized** sibling deployments of the *same protocol* on other VMs (`{vm_family, chain_id, address}` tuples). **Outputs:** one content-addressed *sibling-hypothesis* record per portable sibling deployment — a candidate claim (NOT an executed counterexample) carrying the structural invariant statement, the VM-native runner to confirm it (`bob_sui_run`/`bob_aptos_run`/`bob_anchor_run`/`bob_cosmwasm_run`/`bob_substrate_run`/`bob_foundry_run`), the deployment's ABI fingerprint, and a portability verdict — each also appended to the I7 chain-state-tree as a branch off the confirmed VM-A state so it surfaces in `frontier()` as a leaf to expand. **Predecessors:** I5, I7 (SHIPPED), I13 (Δ-new). **Enforced context budget:** orchestrator `cross_vm_transfer_slice` ≤5k; per-sibling ABI fingerprint ≤800 tokens; per-sibling transfer subagent ≤8k. **Slab:** all transfer records content-addressed (S1); every runner dispatch respects replay-lease typing (S3); no manifest advance without refresh (S2).

## What it does / holds
It stores, per (confirmed-VM-A-invariant × sibling-deployment) pair, a `cross-vm-transfers.jsonl` record keyed by `transfer_hash = hashCanonicalJson({source_run_hash, vulnerability_class, protocol_key, sibling:{vm_family, chain_id, address}})`. Each record holds: the source I5 `run_hash`, the `vulnerability_class`, a **portability verdict** from a frozen `INVARIANT_CLASS_VM_PORTABILITY` table (e.g. `oracle_manipulation` → portable to all six; `signature_validation`/replay → portable with chain-id-binding caveat; `reentrancy` → portable to EVM/CosmWasm, *flagged not-applicable* on Move/Sui because the resource model forbids the callback shape; `delegatecall_storage` → **EVM-only, no sibling emitted**), the sibling deployment's ABI fingerprint (entry functions / capability types / friends from the fetch tools), the matching VM-native runner, and the structural slots still unfilled for that VM. The **bounded query** it enables: `frontier({target_domain})` (`mcp/lib/chain-state-tree.js:178-208`) returns the un-confirmed sibling hypotheses as leaves (default cap 50, hard 500), and the evaluator brief carries a `cross_vm_transfer_slice` of open hypotheses scoped to the smart-contract surface. The **finding a triager recognizes:** "the oracle-spot-price manipulation confirmed on the EVM deployment at `<addr_A>` (Foundry counterexample `run_hash`) reproduces on the protocol's CosmWasm deployment at `<addr_B>` — same price-dependent action, same missing TWAP guard — confirmed via `bob_cosmwasm_run`." Cross-VM provenance (one logic flaw, six possible blast radii) is the distinguishing line no peer can write.

## Failure mode it prevents
A protocol ships the *same* economic logic on EVM and (say) Aptos; a hunter confirms an oracle-manipulation bug on EVM, reports it, and the identical bug on the Aptos deployment goes untouched because (a) no human re-derives the hypothesis across an unfamiliar VM, and (b) the I5 corpus is **Foundry/Solidity-only** (`mcp/lib/invariant-runner.js:385` wraps every template in `forge-std/Test.sol`), so the confirmed invariant cannot mechanically replay on Move/SVM/CosmWasm/Substrate. C16 prevents the silent miss by carrying the *hypothesis* (not a forced Solidity test) across the VM boundary as a scope-gated, auditable lead. Secondarily it prevents the substrate-as-trajectory failure the verifier flagged: six fully-built SC fetch/run stacks (`evm`/`sui`/`aptos`/`svm`/`cosmwasm`/`substrate`) with no Δ-consumer.

1. **Spec the portability table (one commit).** New `mcp/lib/cross-vm-invariant-portability.js`: a frozen `INVARIANT_CLASS_VM_PORTABILITY` keyed by the seven I5 classes (`mcp/lib/invariant-template-corpus.js:5-108`: `reentrancy`, `access_control`, `arithmetic_overflow`, `oracle_manipulation`, `unchecked_call`, `signature_validation`, `delegatecall_storage`). Honest defaults: `delegatecall_storage` → EVM-only; `reentrancy` → omit `sui`/`aptos` (resource model) with a note; `oracle_manipulation`/`arithmetic_overflow` → all six. Runner map: `evm→bob_foundry_run|bob_halmos_run`, `sui→bob_sui_run`, `aptos→bob_aptos_run`, `svm→bob_anchor_run` (SVM execution path = `anchor-runner.js`), `cosmwasm→bob_cosmwasm_run`, `substrate→bob_substrate_run`.

For each sibling: look up portability (slice 1); if not portable, **record a `verdict: "not_portable"` row, never silently drop** (no-premature-termination); if portable, derive unfilled structural slots by re-using `suggestInvariantsForFinding(source_finding)` slot names and diffing against `abi_fingerprints[sibling]`.

## REVIEW (engineering)
- **Tests:** new `test/cross-vm-transfer.test.js` covering: (a) portable class (`oracle_manipulation`) → one sibling record + one I7 branch per authorized deployment; (b) **`delegatecall_storage` → zero siblings, EVM-only verdict recorded** (portability gate); (c) reentrancy → CosmWasm sibling emitted, Sui sibling recorded `not_portable` (never dropped); (d) determinism — same inputs → same `transfer_hash` and same I7 `node_hash`/`state_hash`; (e) scope gate — unauthorized `{chain_id,address}` rejected before any record; (f) malformed/partial `abi_fingerprints` → `unfilled_slots` populated, no throw. Extend `test/chain-state-tree.test.js` only if the branch shape needs a new assertion.
- **Context-budget compliance:** orchestrator `cross_vm_transfer_slice` ≤5k; per-sibling ABI fingerprint ≤800; per-sibling subagent ≤8k. Enforce caps: `frontier` already hard-caps leaves at 500 (`:202`); cap the brief slice at the I1 convention (≤5 default / 25 hard).
- **Determinism/content-hashing:** `transfer_hash` via `hashCanonicalJson` (same primitive as `run_hash` `invariant-runner.js:656` and `node_hash` `chain-state-tree.js:110`); `cross-vm-transfers.jsonl` sorts by hash on write for replay-byte-stability (mirror `writeJsonlNodes` `:49-57`).
- **Failure-mode coverage:** orphan (source `run_hash` not in `invariant-runs.jsonl` → reject, don't fabricate); partial (some siblings unauthorized → process authorized, record rejected as policy-blocked, never half-write); malformed JSONL (reuse strict reader patterns from `invariant-runner.js:114-127`).
- **Slab invariants preserved (cite capability-hypergraph.md:57):** S1 content-addressing (every transfer + branch hashed); S3 replay-lease typing (the VM-native runners are `serialized`/network-allowed mutators — confirmation dispatch must hold the lease); S2 manifest-coupled state (no candidate-claim/cluster advance without refresh); credentials never enter the brief slice.

## REVIEW (parishioner / real-target gate)
**Honest gate is engineering/fixture, long-horizon** (matches topology Δ4 gate at capability-hypergraph-delta.md:131: "C16 transfers an invariant across two VM deployments"). The real-target precondition — a protocol with the *same logic* deployed on ≥2 of the 6 VMs **and** a confirmed invariant on one — is rare in bounty scope, and the non-EVM template corpus does not exist, so C16 cannot auto-execute the sibling. The closing fixture: feed a passed EVM `oracle_manipulation` invariant run + a CosmWasm deployment fixture (real ABI fingerprint shape from `bob_cosmwasm_fetch_contract`) and assert C16 emits a structurally-correct sibling-hypothesis candidate claim, an I7 branch off the EVM state, the correct `bob_cosmwasm_run` runner pointer, and correctly suppresses a `delegatecall_storage` transfer. The aspirational real-target outcome (a triager accepting a cross-VM finding) stays a stretch milestone, not the merge gate.

## Risks & doctrine compliance
- **substrate-as-trajectory:** explicitly cleared — C16 is the *consumer* the verifier said the 6-VM substrate lacked; do not add new fetch substrate, only the transfer routine over the shipped `bob_*_fetch_*` tools.
- **index-without-consumer:** `cross-vm-transfers.jsonl` is consumed by the brief slice (slice 7) + `frontier()` leaves + candidate claims; ship slices 2-7 together, never the store alone.
- **context-as-strategy:** sibling reasoning runs off bounded ABI fingerprints + the portability table, never "dump all six deployments' source into one prompt."
- **auto-kill inverting freeze-then-verify / no-premature-termination:** the confirmed VM-A invariant is frozen (content-addressed `run_hash`) *then* siblings are verified — never the reverse; non-portable/unconfirmable siblings are recorded as `not_portable`/`pending`, **never auto-dismissed** (aligns with the smart-contract surface rule: impact hypothesis before stopping, per `.claude/rules/evaluating.md`).
- **Policy constraints:** the SC fetch tools are read-only, DNS-pinned public RPC with `egress_profile` proxy routing unsupported (per each tool description, e.g. `cosmwasm-fetch-contract.js:54`) — keep cross-VM reads on that path. Plane-O `O-P1 --network none` is **not** in play (this is on-chain RPC, not an OSS container). Hard-rule no-PII: SC addresses are public on-chain identifiers, not operator data — but never inject operator identifiers into any runner dispatch.
- **Frontier safety/scope gate (the load-bearing one):** every sibling `{chain_id, address}` must be independently scope-authorized in the dispatch routine (slice 6) before any fetch or runner call — cross-VM transfer must not become an unauthorized cross-deployment probe.
