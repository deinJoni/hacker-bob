# C10 — Patched-vs-unpatched differential proof
**Kind / Tier / Type:** C / Δ2 / net-new   **Verdict from verification:** keep (WEAVE roll-up: "Net-new differential evidence-pack extension; doctrine-aligned residual proof"). Anchor confirmed; predecessor reframed to S14 (which itself depends on the newly-registered S15 OSS-container substrate).

## Spec (one-paragraph contract)
**Inputs:** a FINAL reportable finding (`F-N`) whose live proof was a `bob_repo_docker_run` row (the *vuln run*, `vuln_run_id`), plus a second `bob_repo_docker_run` row (the *control run*, `control_run_id`) executed against a differential checkout produced by **S14** (`git checkout`/`git apply` on the `read_write` `/work` mount, `--network none`). **Output:** a bounded, content-addressed `differential` object embedded in that finding's evidence pack — `{control_kind, vuln_run_id, control_run_id, control_ref, vuln_fired, control_fired, verdict, control_summary}` — written through `bob_write_evidence_packs` only, mirrored into `evidence-packs.md`, and surfaced in the report snippet. **Predecessors:** S14 (Δ-new). **Unlocks:** C14, C15, C17. **Enforced context budget:** the differential object is scalar-and-enum only (no raw stdout); the on-disk proof lives in `repo-runs/<run_id>.{stdout,stderr}` and is referenced by `run_id` + recomputed `stdout_hash`, never inlined. Caps below.

## What it does / holds
It produces a finding a triager recognizes as **defensible against the standard rebuttal "this is already fixed / not real."** Concretely, the evidence pack for `F-N` now carries a differential row asserting one of three claims, each backed by *two* content-addressed container runs:

- `control_kind: "upstream_fix"` — **residual proof.** The vuln run and the control run are the *same* exploit against a checkout that already contains the maintainer's fix commit; `vuln_fired === control_fired === true` ⇒ `verdict: "residual_confirmed"`. This is the headline (matches the live disclosure campaign's "is the netatalk/openexr fix actually complete?" question, see `project_libraw_disclosure` residual-of-2020-commit pattern).
- `control_kind: "self_patch"` — **patch-fixes-it.** Same exploit, but the control checkout has Bob's proposed fix applied (`git apply`); `vuln_fired === true && control_fired === false` ⇒ `verdict: "patch_fixes"`. (Feeds C15.)
- `control_kind: "pre_introduction"` — **regression localization.** Same exploit run at the parent of a suspected-introducing commit; `vuln_fired === true && control_fired === false` ⇒ `verdict: "regression_localized"` (the bug is absent before commit X).

Any other (fired,fired/not) combination ⇒ `verdict: "inconclusive"` — recorded, never suppressing. Identity of each run is `run_id` + the sha256 of `repo-runs/<run_id>.stdout`, reusing the exact `repo_command_run` evidence-reference contract already validated in `mcp/lib/claims.js:130-152` and projected at completeness time by `projectRepoCommandRunObservedRef` (`mcp/lib/claim-freeze.js:382-395`).

## Failure mode it prevents
A real, recurring loss: a genuine **incomplete-fix / residual** bug gets killed on the rubric "audit reports this fixed" (the precise termination-condition the hunting rules forbid, `.claude/rules/hunting.md` smart-contract clause) or a maintainer closes a report as "already patched" because the PoC ran on an old checkout. Today the only differential affordances are loose and ungated: a free-form `control_expectation` object on a CandidateClaim (`mcp/lib/claims.js:274,304`, no enum, no run binding, no verdict) and an *HTTP* auth-differential runner (`auth-differential-runner.js`) that has no OSS/source analogue. C10 makes "I ran the exploit *with the fix present and it still fired*" a first-class, hash-bound, triager-checkable artifact instead of a prose assertion.

## Predecessors & unlocks
- **S14 — In-container differential checkout (Δ-new, Δ2).** Provides the control checkout via `git checkout`/`git apply` on the `read_write`/`/work` mount (`mcp/lib/repo-env.js:763` `REPO_MOUNT_MODE_VALUES`, `:856` `${workDir}:/work:rw`). **Cross-tier dependency:** S14 in turn rests on **S15 — OSS-container substrate (register, Δ1)**, the node that fixes the systemic S6 mislabel. So the real chain is C10(Δ2) ← S14(Δ2) ← S15(Δ1) — tier-monotone, no inversion. C10 cannot ship before S14 exists, because without an in-container checkout there is no honest way to *produce* `control_run_id`.
- **Unlocks C14 (Δ2)** — the differential row is the load-bearing artifact in the proof-carrying disclosure bundle (`hyperedges.json` H-C10: "substrate for proof-carrying disclosure, self-patch loop, AND the regression sentinel").
- **Unlocks C15 (Δ4)** — the `self_patch` verdict is the loop's success oracle.
- **Unlocks C17 (Δ4)** — the regression sentinel re-fires exactly this `{control_kind, run pair}` shape against HEAD on each new commit.

## Risks & doctrine compliance
- **substrate-as-trajectory:** avoided — C10 ships a real capability (a triager-recognizable residual/regression finding), not a passive store. The `differential` field has named live consumers (C14 bundle, C15 oracle, C17 sentinel, human report render).
- **index-without-consumer:** avoided — the row is rendered in `evidence-packs.md`, folded into the GRADE completeness gate, and consumed by three downstream nodes.
- **context-as-strategy:** avoided — content-addressed by `run_id`/sha, raw output left on disk, all fields capped.
- **auto-kill that inverts freeze-then-verify / no-premature-termination:** **explicitly guarded.** A control that does not reproduce (or any inconsistent fired-pattern) yields `verdict: "inconclusive"` and is *recorded*; it must never remove `F-N` from `finalReportableIdSet`. This is the C15 "a fixed-PoC patch must never auto-suppress the pre-patch finding" invariant applied one tier down, and the cap-not-kill discipline carried over from C9.
- **Policy constraints:** Plane-O **O-P1** — the control run inherits `--network none` (`repo-env.js:1188`; init never clones, `init-repo-session.js:35,38` per WEAVE); the normalizer hard-rejects any control whose JSONL row has `network_mode !== "none"`, so a differential can never be produced by a network-tainted checkout. **Hard-rule no-PII** — `validateNoSensitiveMaterial` runs over the whole `differential` object before persistence, same as every sibling evidence field; the operator's identifiers can never enter `control_summary`/`control_ref`.
- **(assumed)** the exact `network_mode` field name on the persisted `repo-command-runs.jsonl` row is `"none"`/`"bridge"` per `repo-env.js:1188` (read in the dry-run row builder, `:1199`); I did not read the live-mode row writer past `:1244`, so slice 3(c) should confirm the live row carries the same key before keying the reject on it.
