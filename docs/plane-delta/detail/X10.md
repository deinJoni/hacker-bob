# X10 — Reasoning-divergence adjudication

**Kind / Tier / Type:** X (cross-cutting) / Δ2 / harden   **Verdict from verification:** NEW — add (defer-table-safe form of `COMPETITOR_ANALYSIS.md` gap #10; the LLM/semantic-diff form is explicitly out of scope). Pure extension of the shipped verification spine; no slab change.

## Spec (one-paragraph contract)
Given the two independent verification rounds Bob already builds (`brutalist`, `balanced`) inside `buildVerificationAdjudication` (`mcp/lib/verification.js:592`), X10 adds a **deterministic** per-finding signal that detects when both rounds *agree* on `disposition`/`severity`/`reportable` yet their **replay artifacts diverge** — i.e. they likely confirmed two different bugs under one `finding_id`. Input: the two normalized round documents (each result carries `confidence_reasons[]` and `artifact_hashes{}` from `verification-round-store.js:128,135`). Output: a new `reasoning_divergence` enum per finding folded into the adjudication payload (and therefore into `adjudication_plan_hash` via `computeAdjudicationPlanHash`), plus a forced replay on the strong divergence case. Predecessor: `bob_build_verification_adjudication` (SHIPPED). Enforced context budget: **+≤1 enum token per finding** in the compact adjudication context; no new subagent, no new brief slice. **No LLM, no semantic text diff** — comparison is over content hashes only.

## What it does / holds
For every `finding_id` where **both** rounds independently replayed — both results carry `fresh_replay_passed` in `confidence_reasons` (`mcp/lib/constants.js:61`) and both have a non-empty `artifact_hashes` map — X10 compares the two `artifact_hashes` objects deterministically and emits one of:

- `artifact_key_divergence` (**strong**): the two rounds' artifact **key sets are disjoint** — brutalist labelled its replayed artifacts e.g. `{sqli_response}` while balanced labelled `{xss_response}`. Two replays that hold the same finding but describe non-overlapping evidence are the canonical "agree on the row, describe different bugs" case. → recorded **and** forces replay.
- `artifact_hash_divergence` (**weak**): key sets overlap but ≥1 shared key has a different hash value. Recorded for visibility, but **does not** force replay (see failure-mode note — shared-key value drift across two independent live replays is frequently benign nonce/timestamp churn).
- `none`: default; folded in so the field is total over `finding_ids`.

The enum is written into `payload.reasoning_divergence{finding_id→enum}` and `payload.reasoning_divergence_ids[]` (sorted), alongside the existing `disposition_diffs`/`severity_diffs`/`reportable_diffs` arrays at `verification.js:675-678`, and surfaced per-finding in `compactAdjudicationContextFromDocument` (`verification.js:514`) so the verification brief / `bob_read_verification_context` consumer sees it. The triager-recognizable artifact: an adjudication entry reading `reportable=true, agreed disposition/severity, reasoning_divergence=artifact_key_divergence, replay_required` — a one-line, deterministic flag that two reviewers blessed the same row off different evidence.

## Failure mode it prevents
A finding where the brutalist verifier replayed bug A and the balanced verifier replayed bug B, both labelled the row "confirmed / HIGH / reportable," and the current `findingDiffs` (`verification.js:471-473`, which diffs **only** `disposition`, `severity`, `reportable`) sees zero disagreement → the row drops straight into `agreed[]` and ships in the report with two reviewers' apparent consensus over **incompatible evidence**. That mis-attributed/over-stated finding is exactly what a triager closes as "your two PoCs don't match," burning report credibility. X10 forces a re-replay before such a row can be treated as agreed.

## REVIEW (parishioner / real-target gate)

Until a live run produces a both-replayed pair, the gate is satisfied by the seeded integration fixture above; state this as long-horizon, like the I11/X7 calibration gates.

## Risks & doctrine compliance
- **False-positive storm (primary risk, honestly flagged):** two *independent* live replays of the **same** bug routinely produce different artifact hashes (nonces, timestamps, differing-but-valid request shapes). A naive "any hash value differs → divergent" rule would flag nearly every agreed finding. X10 deliberately keys the **strong/replay-forcing** signal on **disjoint key sets** (the metadata *labels* a verifier chose for what it replayed), and demotes shared-key value mismatch to a record-only `artifact_hash_divergence`. This keeps the forced-replay path low-FP and is the core design judgment of the node.
- **Auto-kill / freeze-then-verify:** X10 **never** drops, downgrades, or suppresses a finding. The strong case only *adds* the finding to `replay_required_ids` — a conservative re-verification, fully consistent with no-premature-termination. The finding stays in the reportable set; resolution is human/verifier-gated.
- **substrate-as-trajectory:** not applicable — no new substrate; reuses shipped adjudication state.
- **index-without-consumer:** avoided — the enum has a defined consumer (compact adjudication context → verification brief → `bob_read_verification_context`) and a behavioral effect (forced replay on the strong case). It is not a dangling field.
- **context-as-strategy:** avoided — adds ≤1 token/finding, not a context-width play; the value is a deterministic gate, not more prompt text.
- **Scope / policy:** purely internal verification-spine logic; no target interaction, so Plane-O `--network none` and the hard no-PII rule are unaffected (only hashes of already-recorded, secret-name-rejected artifact keys are compared). No semantic/LLM diff is introduced, keeping X10 inside the defer-table-safe boundary the verification rationale requires.
