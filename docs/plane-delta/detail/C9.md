# C9 — OSS severity-ceiling closure (grade-time lift/cap)

**Kind / Tier / Type:** C (capability) / Δ1 / net-new   **Verdict from verification:** REVISE (WEAVE roll-up line 19, Must-fix #7) — trimmed to the grade-time reachability lift/cap ONLY; chain severity elevation already ships (`prompts/roles/chain.md:8-11,49`) and must not be rebuilt; the AV:L ceiling must CAP severity, never drop a finding from the reportable set.

## Spec (one-paragraph contract)
At grade time, deterministically reconcile each reportable finding's recorded severity against the I9 source-reachability ceiling for the surface(s) it cites, and stamp a `reachability` disposition (`capped` | `lifted` | `unchanged` | `unknown`) onto the grade verdict. **Inputs:** the frozen grade work-set (`readFrozenGradeFindingIdSet`, `grade-verdict-store.js:56-69`), the final verification round's per-finding `{severity, reportable}` (`verification-round-store.js:119-122`), and I9's per-surface reachability (`inventory.reachability.{network_reachable, max_credible_severity_ceiling, network_reachable_surface_ids}` — the shape the dead test at `repo-target.test.js:231-277` asserts) joined via the claim's `surface_ids` (`claims.js:266`). **Outputs:** a per-finding `reachability` object on the grade-verdict document and, on repo sessions only, a soft VERIFY→GRADE blocker when a reportable finding lacks a stamp. **Predecessors:** I9 (Δ-new). **Context budget:** deterministic gate, zero new subagent; the per-finding stamp is ≤5 scalars (~200 tokens, mirroring the brief scalar caps at `assignment-brief.js:139-141`).

## What it does / holds
For each finding in the final-reportable set, it computes a pure function `graded_severity = ceil(recorded_severity, severity_ceiling, attack_vector)`:
- **CAP** — when the reachability ceiling is below the recorded severity (e.g. an AV:L file-parser bug recorded HIGH but living under `local_only_candidate_dirs[]`), `graded_severity` drops to the ceiling. This is the honest-severity correction the evaluator prompt already preaches (`prompts/roles/evaluator.md:65`: "a bug … is AV:L, so record an honest MEDIUM instead of inheriting the surface's CRITICAL") but which nothing enforces at grade time today.
- **LIFT (certify-defensible, not auto-raise)** — when the surface is `network_reachable` and the recorded severity is already HIGH/CRIT, the stamp marks `disposition:"lifted", defensible:true`, certifying the claimed severity is reachability-backed. It does **not** invent severity above what the verification round recorded (that would invert "severity must match verified impact", `reporting.md`). This is the "defensible HIGH/CRIT" output a triager recognizes: a HIGH carrying an explicit AV:N reachability path to a listener.

The finding a triager sees: a grade verdict + report whose per-finding severity equals the reachability-honest `graded_severity`, with the AV:N/AV:L attack-vector and the reachable-surface citation attached — the exact thing that turns the MED/LOW-ceiling pattern (`project_bob_oss_severity_ceiling_diagnosis`) into a defensible HIGH or an honest, un-inflated MEDIUM.

## Failure mode it prevents
Two symmetric bad outcomes: (1) **over-claiming** — a local AV:L parser bug graded and reported as HIGH, which a maintainer rejects on sight and which burns the channel (`feedback_oss_severity_honest_gating`); and (2) **under-grading** — a genuinely network-reachable OOB-write left at MEDIUM because no grade-time signal certifies its AV:N reachability, leaving real CRITICALs filed as MED/LOW. The hard guard it additionally prevents: the cap being weaponized into an **auto-kill** that silently removes a finding from the reportable set (a freeze-then-verify inversion).

## Risks & doctrine compliance
- **Auto-kill / freeze-then-verify inversion (primary risk):** the cap must change only the `graded_severity` label, never reportable-set membership. Enforced by leaving the membership filter (`grade-verdict-store.js:227-231`) on recorded severity and by a dedicated regression test. This is the literal "cap, never kill" mandate (WEAVE #7, node `verify`).
- **Lift must not invent severity (reporting-doctrine):** the lift direction only *certifies* an already-recorded HIGH/CRIT as reachability-defensible; auto-raising a MEDIUM to HIGH from reachability alone would violate "severity must match verified impact" (`reporting.md`) and is explicitly out of scope.
- **Index-without-consumer (inverted):** C9 *is* I9's grade-time consumer, closing the inert-consumer regression — but it must ship after/with I9, or it becomes a consumer-without-producer. Strict ordering flagged.
- **Context-as-strategy / substrate-as-trajectory:** none — no new brief width, no new subagent, no new slab dir; purely deterministic reconciliation over existing artifacts.
- **Plane-O O-P1 (`--network none`, init-never-clones):** inherited transitively via S15→I9; C9 itself performs no network or container I/O, so it does not touch O-P1.
- **Hard-rule no-PII:** the stamp carries only file paths, severity labels, and an attack-vector enum — no operator identifiers ever enter the grade verdict.
- **Cross-tier dependency (must decide before locking the tier):** the maintainer-accepted-HIGH milestone couples to C12/Δ3; recommend the "soften the Δ1 gate to stamp-present-+-consumed-+-lifted/capped" option over widening C9 into the record-time proof-gate path, to keep C9 a clean Δ1.
