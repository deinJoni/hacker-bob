# X7 — Evaluator-run-avoided telemetry

**Kind / Tier / Type:** X (cross-cutting) / Δ2 / adopt   **Verdict from verification:** SPLIT — the per-session *descriptive count* stays here as a Δ2 ride-along that needs **neither S11 nor I11**; the `$`-cost / LLM-economics half moved to **X7b (Δ4)** because it needs a token-telemetry substrate Bob lacks. The "descriptive-only, never an auto-kill gate" guard is the load-bearing constraint and is preserved verbatim below.

## Spec (one-paragraph contract)
**Inputs:** the lead set already in scope at every real promotion call (`document.leads` in `promoteSurfaceLeadsInternal`, `mcp/lib/lead-promotion.js:317-341`) plus the resolved `{limit, minScore, includeMedium}` options (`normalizePromotionOptions`, `mcp/lib/lead-scoring.js:70-75`). **Output:** exactly one new append-only `evaluator_run_avoided` pipeline event per promotion call, carrying an integer `counts` block (assignable / promoted / filtered / deferred_by_limit / evaluator_runs_avoided), surfaced read-side in the `analyzeSession` row and the bob-status (dashboard) panel. **Predecessors:** `lead-scoring.js` (SHIPPED — `shouldPromoteLead`), `pipeline-events.js` (SHIPPED — event substrate + frozen type enum), `pipeline-analytics.js` + `dashboard.js` (SHIPPED — consumers). **Enforced context budget:** the event is operator-facing telemetry ONLY — it is never injected into any evaluator/hunter brief slice (brief budget impact = 0); the event body stays within the existing `capString`/`normalizeCounts` caps (`pipeline-events.js:62-99`) and the read-side row gains ≤6 integer fields (<200 tokens). **Hard contract:** X7 is a pure *observer* of the `shouldPromoteLead` decision that already happened — it adds zero new filtering and must never feed back into a promotion or kill decision.

## What it does / holds
At each real promotion (`promoteSurfaceLeadsInternal`, the shared core behind the orchestrator tool `bob_promote_surface_leads` → `promoteSurfaceLeads`, `mcp/lib/lead-promotion.js:380-385`, and the wave auto-path `promoteSurfaceLeadsForWave`, `:388-391`), `selectPromotableSurfaceLeads` (`lead-scoring.js:82-87`) already partitions `document.leads` via `shouldPromoteLead` (`lead-scoring.js:61-68`) and then slices to `limit`. X7 computes — from that same already-made decision — the descriptive tuple:

- `assignable` = leads where `isAssignableSurfaceLead(lead)` (`lead-scoring.js:54-59`) and not already promoted (`lead.status !== "promoted" && !lead.promoted_surface_id`, the same guards as `shouldPromoteLead:62`);
- `promoted` = `candidates.length` after the limit slice (the leads that actually become frontier surfaces);
- `filtered` = assignable leads that `shouldPromoteLead` rejected (low score / not high-confidence / medium excluded);
- `deferred_by_limit` = assignable leads that *passed* the predicate but were cut by `limit` (default 8, `lead-scoring.js:71`);
- `evaluator_runs_avoided` = `filtered` — the honest estimate, since every promoted lead becomes one frontier surface that a later wave assigns to roughly one evaluator subagent, so each filtered assignable lead is one evaluator subagent **not** dispatched.

It emits these as an `evaluator_run_avoided` event using the canonical best-effort producer pattern already used by `coverage.js:280-290`: `safeAppendPipelineEventDirect(domain, "evaluator_run_avoided", { source, counts }, safeGovernanceContextForDomain(domain))` (`pipeline-events.js:294-300`, `governance-context.js:89-98`). Read-side, `analyzeSession` sums these events (same shape as the `finding_index_failed` scan at `pipeline-analytics.js:653`) into a `lead_promotion` block on the session `row` (`:814-883`), and the dashboard panel (`dashboard.js:300`) renders "N evaluator runs avoided this session." The **bounded query it enables**: "how many evaluator subagents did lead-scoring save me this session / across recent sessions, and is my `min_score`/`include_medium` tuning too aggressive?" — answerable without re-reading any lead document.

## Failure mode it prevents
The concrete bad outcome: Bob's lead gate **already** silently discards low-value leads (saving evaluator spawns and tokens), but **nothing records that it happened**, so the operator tuning the breadth-vs-cost dial (the exact `min_score 60` / `include_medium false` knobs in `normalizePromotionOptions:70-75`, cf. the max-thorough tuning work) is flying blind: they cannot see whether the gate is dropping 2 leads or 200, cannot justify the savings, and cannot detect a misconfigured `min_score` that quietly starves waves. X7 makes the existing filtering *measurable* — turning an invisible behavior into a tuning signal — without changing one bit of promotion behavior.

- `lead-scoring.js` — **SHIPPED**. `shouldPromoteLead`, `isAssignableSurfaceLead`, `selectPromotableSurfaceLeads`, `normalizePromotionOptions`. X7 reads its decision; does not modify it.

- **Cross-tier dependency:** none. The whole point of the SPLIT is that X7 stays Δ2 and depends on **neither S11 (Δ4) nor I11 (Δ4)**. No upward edge.

## Context budget
No subagent and no brief slice are introduced, so there is no per-subagent cap to set. Concrete caps: event body ≤ `source` 120 chars + `counts` ≤6 integer keys (`normalizeCounts`); read-side `lead_promotion` block ≤6 integer fields (<200 tokens) on a row that is already operator-facing; JSONL bounded at `PIPELINE_EVENTS_MAX_RECORDS = 5000` (`pipeline-events.js:26`). Evaluator/hunter-brief budget impact: **0** (telemetry is never threaded into a brief).

## Risks & doctrine compliance
- **Auto-kill that inverts freeze-then-verify / no-premature-termination (the primary risk):** X7 must remain a *read-only observer* of the existing `shouldPromoteLead` decision. It introduces no new predicate and must never feed back into promotion, wave-sizing, or any kill/skip gate — that would make it a sibling of the doctrine-rejected `X-rej-1`/`X-rej-2` and could suppress a real 0-day hiding in a low-scored lead. The spec hard-codes descriptive-only and emits *after* the decision, never before it.
- **Index-without-consumer:** avoided by shipping the consumers in the same work (analytics row + dashboard panel, DO slices 6–7). The event is not a dangling producer.
- **Substrate-as-trajectory / context-as-strategy:** the count is a bounded integer telemetry record, deliberately kept out of every brief; it does not widen any agent's context.
- **Goodhart / metric-as-target:** "runs avoided" is descriptive, not a KPI promotion optimizes toward; it must stay decoupled from `min_score`/`include_medium` so tuning those knobs is the operator's call, informed by — never driven by — the number.
- **Policy constraints:** no network egress, no scope interaction (`network_access:false`, `scope_required:false` on the tool, `tools/promote-surface-leads.js:25-27`); dashboard surface is loopback/unauthenticated (`dashboard.js:575`) and the payload contains no PII (hard-rule no-PII satisfied — counts only). No Plane-O `--network none` interaction (this is the web/evaluate lead path, not the OSS container substrate). No frontier safety/scope gate applies (X7 is non-frontier, Δ2, observer-only).
