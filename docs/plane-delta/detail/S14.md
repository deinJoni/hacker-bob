# S14 — In-container differential checkout
**Kind / Tier / Type:** S (Slab) / Δ2 / net-new   **Verdict from verification:** revise (✏️) — original "init-repo-session full-clone flag" anchor REJECTED: it violates Plane-O O-P1 (init never clones — `mcp/lib/tools/init-repo-session.js:31,38`; `mcp/lib/repo-target.js:11`) and `--network none` (`mcp/lib/repo-env.js:838`) makes in-container fetch impossible. Reframed as a net-new in-container `git`-materialization codepath on the already-rw `/work` mount + a host-side shallow-clone GUARD; effort raised off "adopt".

## Spec (one-paragraph contract)
**Inputs:** a bound OSS repo session (`bob_init_repo_session`), a `checkout_ref` (7–64 hex commit or local ref) and a `checkout_kind ∈ {upstream_fix, pre_introduction, self_patch}`. **Outputs:** a deterministic working tree of `checkout_ref` materialized into the container's writable `/work/repo` (host `repoWorkDir(domain)`, `mcp/lib/paths.js:332`) **without** mutating the read-only `/src` bind, plus a content-addressed `repo-command-runs.jsonl` row stamped with `{checkout_ref, checkout_kind}` provenance so C10 can bind a `{vuln_run_id, control_run_id}` pair. **Predecessors:** S15 (Δ-new, register of the OSS-container substrate). **Unlocks:** C10. **Enforced context budget:** no new subagent and no new brief slice — provenance is two short scalar fields on the existing JSONL row; command tokens stay ≤ `REPO_DOCKER_RUN_MAX_TOKEN_LENGTH` (2048, `repo-env.js:753`), capture ≤ 16 MB/stream (`repo-env.js:751`), first-line surface ≤ 200 chars (`repo-env.js:758`).

## What it does / holds
It adds the one git operation the Plane-O substrate deliberately omits: materializing a *different* in-history commit into a buildable tree, using only objects already present in the operator's local `/src/.git`, under `--network none`, without writing to `/src`. Concretely it composes a command for `bob_repo_docker_run` of the form `git -C /src cat-file -e <ref>^{commit} && git -C /src archive --format=tar <ref> | tar -x -C /work/repo` — pure-read on the `:ro` `/src` mount (`repo-env.js:855`), pure-write on the always-`:rw` `/work` mount (`repo-env.js:856`), zero network. For `self_patch` it archives `HEAD` into `/work/repo` then `git -C /work/repo apply /work/patch.diff` against a patch staged earlier into `/work` (no network). The guard holds the invariant that *the needed history is local before any differential is attempted*: if `<repoRoot>/.git/shallow` exists, or the ref's object is absent, it refuses loudly rather than silently degrading to a single-tree run. As a slab it provides the C10 capability the property "the control tree was built from the same local object store as the vuln tree, content-addressed by `argv_hash`/`command_hash`."

## Failure mode it prevents
Two bad outcomes. (1) The policy-violating one the original anchor would have caused: re-introducing a remote clone/fetch into init or opening the network inside the sandbox to obtain the fix commit — breaking O-P1 and `--network none`. (2) The silent-false-negative one: running a "patched-vs-unpatched" differential against a **shallow** local checkout where the fix/pre-introduction commit simply isn't in `/src/.git`. Without the guard, `git archive` fails or yields an empty/HEAD tree, the control build produces nothing, and C10 records a bogus "fix makes the bug disappear" verdict — manufacturing a fake residual or fake-resolved finding. The guard converts that into an explicit `shallow_clone_blocks_differential` / `ref_not_in_local_history` refusal with the remediation "deepen the operator's local clone; the framework will not clone for you."

## REVIEW (parishioner / real-target gate)
The honest real-target outcome that *fully* closes S14 is realized **through C10**, not S14 alone: on a fully-cloned local checkout of a real OSS target carrying a known-fixed CVE, S14 materializes the `upstream_fix` commit into `/work/repo`, the same PoC is run against both trees, and the recorded differential demonstrates a *residual* (fix present, bug still fires) that rebuts a maintainer's "already fixed" — submission-grade to that project's disclosure channel. For S14 **in isolation** the gate is engineering/fixture (and explicitly long-horizon until C10 ships): on a two-commit fixture git repo (bug commit + fix commit), the builder materializes each commit into `/work` and a stubbed build/run yields two distinct content-addressed `run_id`s; a `git clone --depth=1` fixture is refused with `shallow_clone_blocks_differential`. State this dependency — do not claim parishioner value for S14 absent its C10 consumer.

## Risks & doctrine compliance
- **Substrate-as-trajectory (anti-pattern #1) / substrate-without-consumer (#4):** S14 is a slab node and the strongest temptation here is to ship it standalone. Mitigation: S14 lands **in the same Δ2 PR/tier as C10**, its only consumer — the differential row is the pull. Do not merge S14 alone.
- **Context-as-strategy (#2):** differential proof is two bounded, content-addressed run records, never a widened context window.
- **Auto-kill / freeze-then-verify / no-premature-termination:** the guard must **fail loud** (structured refusal), never silently skip the control build and let a stale "fixed" disposition stand. S14 only *materializes and records*; it must not itself emit a resolved/residual verdict — that judgment is C10/C15. An empty or failed control build is surfaced as a run failure, not auto-interpreted as "bug is gone."
- **Plane-O O-P1 (policy constraint):** the guard performs zero clone/fetch; `bob_init_repo_session` is untouched; `--network none` is preserved on every materialization run. The only git invocations are read-only (`cat-file`, `archive`) inside the existing sandbox on the `:ro` `/src` mount.
- **Hard-rule no-PII:** `checkout_ref` is a hex commit/local ref; operator personal identifiers can never enter it, and the O-P7 `validateNoSensitiveMaterial` scrub (`repo-env.js:1212`) runs over the provenance-stamped row before persistence.
