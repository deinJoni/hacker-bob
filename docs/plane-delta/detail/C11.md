## Spec (one-paragraph contract)
C11 is the orchestration routine + OSS brief slice + lead-source wiring that turns three indexes into a single ranked, reachability-aware source-audit queue. **Inputs:** I10 static-analysis findings (SARIF, semgrep/trivy via `bob_repo_docker_run`), I9 per-surface reachability (`severity_ceiling` / `attack_vector` / `network_reachable`, restored from blob `d1647c0`), and I12 root-cause families. **Output:** a set of surface leads recorded through the *existing* `recordSurfaceLeadsInternal` path (`mcp/lib/lead-promotion.js:270`) with `source: "bob_static_scan"` (already whitelisted at `lead-promotion.js:60`), scored so a static finding on a network-reachable sink clears the `min_score` 60 gate (`lead-scoring.js:72`) and a local-only one is deprioritized but not dropped — plus a new `static_analysis_leads` slice in `OSS_BRIEF_SLICE_REGISTRY` (`assignment-brief.js:393`) the evaluator reads to drive source→sink tracing. **Predecessors:** I10, I9, I12 (all Δ-new; heaviest fan-in in the graph). **Unlocks:** `findings`. **Enforced context budget:** the new OSS slice is capped at 4096 chars (mirroring web's `static_scan_hints` slice at `assignment-brief.js:276`), top-10 ranked leads (mirroring `STATIC_SCAN_HINT_MAX_ITEMS = 10`, `constants.js:99`).

## What it does / holds
C11 holds no new persistent store of its own — it is a *consumer/composer* over I9/I10/I12. Concretely it does three things:

1. **Source→lead mapping.** Each I10 SARIF result `(rule_id, file, line, message, CWE)` is mapped onto the existing surface-lead shape (`lead-intake.js:67`): `endpoints = ["<file>:<line>"]`, `bug_class_hints = [<family from I12>]`, `surface_type = "oss_static_sink"`, `title = <rule_id>`, recorded via `recordSurfaceLeadsForWaveHandoff` / `recordSurfaceLeads` (`lead-promotion.js:343,358`). These flow through the unchanged frontier dual-write (`emitFrontierEnqueued` → `frontier.enqueued`, `lead-promotion.js:209`) so the materializer folds them like any other lead.
2. **Reachability-weighted scoring.** A scoring augment lifts a lead's score when I9 marks its containing surface `attack_vector: "network"` / `network_reachable: true`, and caps (does not zero) it when `attack_vector: "local"`. This is what makes a reachable sink clear the `min_score` 60 promotion gate (`lead-scoring.js:61-72`) while a local-only sink lands sub-threshold with a rationale (so `enforceLeadRationalePolicy`, `lead-promotion.js:242`, and the Fix #134 soft-warning, `:85`, both stay satisfied).
3. **Brief rendering + lens prose.** A new `static_analysis_leads` slice renders the top-10 reachable leads (file:line, family, AV, recommendation) into the OSS brief, and the evaluator OSS-lens prose (`prompts/roles/evaluator.md:56-66`) tells the agent to *start* from this ranked queue, trace source→sink, and prove the highest-AV ones via real `bob_repo_docker_run` replay (O-P4 gate, `evaluator.md:62`).

**The finding a triager recognizes:** "Attacker-controlled `<file>` input reaches `<unsafe sink>` at `file:line` via `<function>`; static analysis flagged it, reachability confirms an unauthenticated `AV:N` path through `<listener>`, replayed with `bob_repo_docker_run` (exit/ASAN evidence)." That is a maintainer-actionable HIGH, not a CWE-class lint dump.

## Failure mode it prevents
The OSS evaluator reading a large codebase unguided, or being handed a flat 400-line semgrep/trivy dump with no reachability or family signal, and burning its wave chasing unreachable or local-only sinks — yielding the MED/LOW severity ceiling diagnosed in `project_bob_oss_severity_ceiling_diagnosis`. It also prevents the *inverse* silent-drop regression: a sink that static analysis flagged AND I9 confirmed reachable being lost because the OSS brief registry today has **no slice that carries static-analysis leads** (`OSS_BRIEF_SLICE_REGISTRY`, `assignment-brief.js:393-410`, has `repo_workflow / governance / goal_orientation / code_surface_pack / technique_packs / cli_tools / recap_and_handoff` only — no `static_scan_hints`, no reachability slice). That is exactly the orphaned-consumer class that already bit I9 (producer dropped, consumers inert) and `ossTechniquePacks` (`assignment-brief.js:403`, consumer with no producer).

## Risks & doctrine compliance
- **substrate-as-trajectory:** the dominant risk. The Δ3 tier gates on real findings (`capability-hypergraph.md` "Tiers gate on parishioner review"); building the SARIF→lead pipe and stopping is failure. Mitigation: the parishioner gate above is a real-target maintainer-accepted HIGH, not a pipeline-exists check.
- **index-without-consumer:** C11 *is* the consumer that justifies I10/I12; but C11's own consumed reachability fields must be registered (DO slice 5) or the next refactor silently re-drops them — exactly the I9 regression (`WEAVE.md` headline). Enforced via the stigmergy-coherence gate.
- **context-as-strategy:** do not dump all SARIF findings into the brief. Rank by reachability×family, cap at 10/4096 (DO slice 4). The whole point is selectivity, not "stuff every analyzer hit in context."
- **auto-kill that inverts freeze-then-verify / no-premature-termination:** the reachability scoring must **cap/deprioritize** a local-only or unreachable static lead, **never auto-suppress** it (mirrors C9's cap-not-kill mandate, `WEAVE.md` #7). A static lead is a *hypothesis*; the evaluator must verify it live via docker replay before it becomes a finding — the lead is frozen as a low-score, rationale-bearing record, not deleted.
- **Plane-O policy:** `bob_repo_docker_run` defaults `--network none` + `repo_mount_mode: read_only` (O-P3, `evaluator.md:59`); never auto-promote a source finding to the deployed sibling instance (O-P2, `evaluator.md:57`); HIGH/CRITICAL native claims require real replay (O-P4, `evaluator.md:62`). C11 introduces no network egress and no new tool surface.
- **Hard rule — no operator PII as tool input:** N/A to the lead-record path (OSS local checkout, redacted source bytes only), but the record path must never carry operator identifiers; the existing redaction boundary (`redactStaticArtifactContent`) plus `normalizeOptionalString` clamping cover this. No `bounty_temp_email`/signup surface is touched.
- **Frontier-node safety/scope gate:** not a frontier node (no cross-domain/never-before-seen autonomy); standard scope gate is the in-scope repo checkout. No additional safety gate required beyond the Plane-O policy above.
