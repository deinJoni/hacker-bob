# I10 — Static-analysis finding index (SARIF)

**Kind / Tier / Type:** I (pillar/index) · Δ3 · net-new   **Verdict from verification:** keep (WEAVE roll-up: "Net-new — `bob_static_scan` is token-contract only; zero SARIF footprint. Clean index→C11."). Absorbs IP7's SARIF-ingest half (verify-IP.md:48 "seriously consider merging IP7 into I10"); the runner half already ships.

## Spec (one-paragraph contract)
**Inputs:** the SARIF (or trivy-JSON) output captured to `repo-runs/<run_id>.stdout` by an in-container static-analysis run dispatched through `repoDockerRun` (`mcp/lib/repo-env.js:1073`) — i.e. `semgrep --sarif` / `trivy fs --format sarif` seeded by `cli-tool-packs.js:219-232`, plus (Δ-new tool seeds) CodeQL/Coccinelle which are currently "Deferred" (`cli-tool-packs.js:213`). **Outputs:** a deduped, content-hashed, sensitive-material-scrubbed JSONL index (`static-analysis-index.jsonl` under `sessionDir`) and a bounded `queryStaticAnalysisIndex(domain, …)` plus a SARIF→surface-lead mapping that emits `status:"new", promote:false` **LEADS only — never auto-findings**. **Predecessors:** IP7 (SARIF parse, Δ-new), S15 (registered OSS-container substrate, Δ-new register-node). **Unlocks:** C11 (static-driven source audit); augments I9. **Enforced context budget:** parse/dedup/scrub is fully deterministic and **non-LLM** (zero token cost); the only token surface is C11's brief slice, capped (see Context budget).

## What it does / holds
**Stores:** one record per *distinct* static-analysis result, keyed by a content hash over `(tool, ruleId, normalized file path, region.startLine, normalized message)`. Each record holds: `tool` (`semgrep|trivy|codeql|coccinelle`), `rule_id`, `severity` (SARIF `level` → `error|warning|note`), `location` (`path:line:col`), a **scrubbed** message snippet, `cwe`/`tags` from `rule.properties`, `dataflow_steps` count (CodeQL `codeFlows`, when present), and `run_id`/`stdout_hash` provenance back to the `repoCommandRunsJsonlPath` row.
**Bounded query it enables:** `queryStaticAnalysisIndex(domain, { top_k, min_severity, rule_id?, surface_id? })` → ranked compact records (severity-then-dataflow-depth), so C11 opens with "the N static hits on my assigned surface" instead of re-parsing megabytes of SARIF. The companion `mapSarifResultToSurfaceLead()` produces a `normalizeSurfaceLead`-shaped object (`lead-intake.js:67`): `source:"static-analysis:<tool>"`, `bug_class_hints:[rule_id/cwe]`, `evidence:["path:line — <scrubbed message>"]`, `confidence` capped at `medium` for `error` / `low` otherwise (a static hit is *unverified* — it must never enter as `high`), and a `rationale` (rule id + why surfaced) so it survives `enforceLeadRationalePolicy` (`lead-promotion.js:270+`). C11 then routes these through `recordSurfaceLeadsForWaveHandoff` (`lead-promotion.js:358`), and the evaluator that picks up the lead produces the finding a triager recognizes.

## Failure mode it prevents
Two concrete bad outcomes. (1) **Static analysis that goes nowhere:** today `semgrep`/`trivy` run in-container, but their output is captured to a 16 MB stdout file (`repo-env.js:751`) and *nothing parses it* — `grep -rni sarif mcp/ scripts/ test/` is empty. The signal is produced and discarded. I10 turns that raw capture into a deduped, queryable index a hunter can act on. (2) **Static hits masquerading as findings:** without an index that forces `status:"new", promote:false`, an agent could treat a semgrep/CodeQL hit as a confirmed bug. I10 makes every static result a *lead requiring live verification*, preserving freeze-then-verify. A third, security-specific failure: `trivy --scanners secret` SARIF can contain real leaked secret values — I10 scrubs through the existing sanitizers before persistence so the index never becomes a credential cache.

## Risks & doctrine compliance
- **substrate-as-trajectory:** clean — I10 is a passive index queried on demand; it never drives control flow or auto-schedules work.
- **index-without-consumer:** mitigated by design — C11 is the in-tier (Δ3) consumer and the I9 augment edge is a second consumer; slice 7 registers producer+consumer in the stigmergy manifests so CI fails if the consumer is ever dropped (the X9 lesson).
- **context-as-strategy:** mitigated — token cost is bounded to C11's ≤2k slice; the heavy parse is non-LLM.
- **auto-kill / freeze-then-verify inversion:** explicitly preserved — every SARIF result becomes a `status:"new", promote:false` lead with `confidence ≤ medium`; I10 **never** records a finding, suppresses, or auto-promotes. No `excluded_by_doctrine` drift: I10 emits leads, not auto-findings (verify-C.md:22), and it is on the right side of X-rej-5 (it ingests *source-SARIF text*, not a compiled binary-oracle) and X-rej-1 (no auto-skip).
- **Policy constraints:** Plane-O O-P1 holds — the static-analysis runs are in-container with default `--network none` (`repo-env.js:838`); CodeQL needing a build runs against the read-only `/src` (or a `/work` staging mount, `repo-env.js:763,856`), never a network clone. Sensitive-material scrub satisfies the no-credential-leakage slab invariant and O-P7. Hard-rule no-PII is upheld: the index persists only scrubbed code locations, never operator identifiers.
